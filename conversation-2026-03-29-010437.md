  二、陀螺仪工作原理

  文件：rmcs_core/src/hardware/device/bmi088.hpp

  原始数据转换

  int16_t 原始值
    → gyro_rad/s = raw / 32767.0 * 2000.0 / 180.0 * π
    → accel_m/s² = raw / 32767.0 * 6.0

  Mahony AHRS 姿态融合

  BMI088 只有陀螺仪+加速度计（无磁力计），使用 Mahony 互补滤波融合：

  每1ms调用 update_status()

  1. 加速度计归一化 → 估计"重力方向"
  2. 与当前四元数预测的重力方向做叉积 → 得到误差向量 e
  3. 误差反馈:
     积分项(Ki=0): 校正陀螺零偏 (本项目Ki=0，不用积分)
     比例项(Kp=0.2): halfvx/y/z += e * Kp/2
  4. 用修正后的陀螺仪角速度积分更新四元数:
     q0 += (-q1*gx - q2*gy - q3*gz) * dt/2
     q1 += ( q0*gx + q2*gz - q3*gy) * dt/2
     q2 += ( q0*gy - q1*gz + q3*gx) * dt/2
     q3 += ( q0*gz + q1*gy - q2*gx) * dt/2
  5. 四元数归一化

  物理含义：陀螺仪积分给出高频精确姿态，加速度计给出低频重力方向修正，二者融合得
  到稳定的绝对姿态四元数 q = [q0, q1, q2, q3]。

  速度输出

  // omni_infantry.cpp:244-245
  *gimbal_yaw_velocity_imu_   = bmi088_.gz();  // 绕Z轴角速度 = yaw速度
  *gimbal_pitch_velocity_imu_ = bmi088_.gy();  // 绕Y轴角速度 = pitch速度

  这是最核心的部分。解决的问题是：给定一个世界坐标系中的目标方向，如何分解成独立

  SetControlShift（手动控制）

  pitch_rotation = AngleAxis(pitch_shift, YawLink::Y cast to OdomImu)

  为什么在 OdomImu 帧操作？ 因为操作者（人或自瞄算法）的意图是在绝对空间中移动云
  台，而不是在随底盘运动的局部坐标系中。这样底盘旋转不会干扰云台的绝对指向。

  Yaw 轴滤波（update_yaw_axis）

  // 低通滤波 YawLink 的 Z 轴在 OdomImu 中的表达
  yaw_axis = yaw_axis * 0.9 + new_yaw_axis * 0.1

  防止底盘剧烈运动时 yaw 轴跳变。

  pitch_link_to_yaw_link / 反变换

  控制方向 (OdomImu帧中的3D向量)
      │ fast_tf::cast<OdomImu, YawLink>()
      ↓
  YawLink 坐标系中的方向向量 (x,y,z)
      │
      ├─ yaw_angle_error  = atan2(y, x)          ← 水平偏差
      └─ pitch_angle_error = -atan2(z*c - √(x²+y²)*s,
                                    z*s + √(x²+y²)*c)  ← 俯仰偏差

  pitch 公式的几何含义：把3D方向向量投影到 YawLink 坐标系后，√(x²+y²)
  是水平分量，z 是垂直分量。考虑到 pitch 电机当前已经转了一定角度，需要用当前
  pitch 角的三角函数旋转基底，才能得到正确的"还差多少角度"。

      │

  第一章：基础概念（先把工具搞懂）

  在看代码之前，必须先理解几个基础概念。

  1.1 什么是坐标系（Frame / Link）

  想象你站在房间里，你可以用"房间坐标系"描述位置（东西南北）。但如果你转了个身，
  你身体的"前方"就不再是固定的东了。这就是两个不同的坐标系。

  在这个代码里，每个刚性部件都有自己的坐标系：

  BaseLink        → 底盘坐标系（跟底盘一起转）
  GimbalCenterLink → 云台安装座坐标系（固定在底盘上）
  YawLink         → yaw电机转动后的坐标系
  PitchLink       → pitch电机转动后的坐标系
  OdomImu         → 世界惯性坐标系（不随任何运动而变化）

  直觉理解：你的头就是云台。你的脖子左右转是 yaw，上下点头是
  pitch。你的身体是底盘。而"世界的北方"就是
  OdomImu，不管你怎么转，北方永远是北方。

  1.2 什么是向量

  向量就是"一个有方向的箭头"。[1, 0, 0] 表示沿 X 轴方向的单位向量。

  关键点：同一个箭头（比如枪管指向），在不同坐标系里看，数值是不同的。这就是为什
  么需要坐标变换。

  1.3 什么是四元数

  四元数是描述"旋转"的数学工具，由 4 个数 [w, x, y, z]
  组成。你不需要理解它的数学，只需要知道：

  - 它能精确描述三维空间中的任意旋转
  - 它比欧拉角（俯仰偏航滚转）更稳定，不会有万向节死锁问题
  - quaternion.conjugate() 就是"这个旋转的逆"，即转回去

  第二章：陀螺仪（BMI088）工作原理

  文件：rmcs_core/src/hardware/device/bmi088.hpp

  2.1 BMI088 是什么

  BMI088 是一块芯片，里面有两种传感器：
  - 陀螺仪（Gyroscope）：测量转动速度（单位：rad/s，弧度/秒）
  - 加速度计（Accelerometer）：测量加速度（单位：m/s²）

  陀螺仪的问题：它只能测角速度，要知道当前角度就要对速度积分（累加）。积分会累积
  误差，时间长了会漂移——就像闭眼走路，走一段后就不知道朝哪了。

  加速度计的问题：它能感受重力方向（告诉你"哪里是下面"），但只有静止时才准确，运
  动时会被加速度干扰。

  解决方案：把两者融合——用加速度计的"重力方向"来不断纠正陀螺仪的漂移。这就是
  Mahony 互补滤波算法。

  2.2 原始数据转换（第40-57行）

  // bmi088.hpp:44-46
  auto solve_acc  = [](int16_t value) { return value / 32767.0 * 6.0; };
  auto solve_gyro = [](int16_t value) {
      return value / 32767.0 * 2000.0 / 180.0 * std::numbers::pi;
  };

  传感器给出的是 16 位整数（-32767 到 +32767），需要换算成物理量：
  - 加速度：除以32767，再乘以6 → 范围是 ±6 m/s²
  - 角速度：除以32767，再乘以2000（度/秒），再除以180，再乘π → 换算成
  rad/s，范围是 ±2000°/s

  // bmi088.hpp:49-54
  gx_ = solve_gyro(gyro.x), gy_ = solve_gyro(gyro.y), gz_ = solve_gyro(gyro.z);
  ax_ = solve_acc(acc.x),  ay_ = solve_acc(acc.y),  az_ = solve_acc(acc.z);

  if (coordinate_mapping_function_) {  // 坐标轴重映射（如果芯片装歪了）
      std::tie(gx_, gy_, gz_) = coordinate_mapping_function_(gx_, gy_, gz_);
      std::tie(ax_, ay_, az_) = coordinate_mapping_function_(ax_, ay_, az_);
  }

  coordinate_mapping_function_ 是用来处理"传感器装歪了"的情况——如果 BMI088
  在电路板上是倒装的，X/Y/Z 轴就需要重新映射。

  2.3 Mahony 姿态融合算法（第74-144行）

  这是整个 IMU 的核心，分四步：

  步骤1：用当前四元数预测重力方向（第94-96行）

  // bmi088.hpp:94-96
  halfvx = (q1_ * q3_) - (q0_ * q2_);
  halfvy = (q0_ * q1_) + (q2_ * q3_);
  halfvz = (q0_ * q0_) - 0.5 + (q3_ * q3_);

  直觉：如果我现在的姿态是四元数 q，那么"世界向下的方向 [0,0,-1]"
  在机体坐标系中应该是什么方向？这三行就是在做这个计算（省略了一半，因此叫
  half）。

  步骤2：计算误差（第99-101行）

  // bmi088.hpp:99-101
  halfex = (ay * halfvz) - (az * halfvy);
  halfey = (az * halfvx) - (ax * halfvz);
  halfez = (ax * halfvy) - (ay * halfvx);

  直觉：
  - halfvx/y/z 是"按当前四元数预测的重力方向"
  - ax/ay/az 是"加速度计实际测到的重力方向"（已归一化）
  - 叉积（cross product）的结果就是"这两个方向的偏差"

  叉积可以理解为：两个向量叉乘，结果是垂直于两者的向量，大小等于它们夹角的正弦值
  。偏差越大，叉积越大。

  步骤3：用误差修正陀螺仪（第120-123行）

  // bmi088.hpp:120-123
  gx += double_kp_ * halfex;  // double_kp_ = 2 * 0.2 = 0.4
  gy += double_kp_ * halfey;
  gz += double_kp_ * halfez;

  直觉：把误差乘以增益 Kp，加到陀螺仪读数上。这相当于"在陀螺仪的测量值里，叠加一
  个修正量，让姿态朝着加速度计指示的方向靠拢"。

  Kp=0.2 是一个调参结果：太大则加速度计噪声进来太多，太小则漂移修正太慢。本项目
  Ki=0，积分项被禁用。

  步骤4：用修正后的角速度积分四元数（第127-143行）

  // bmi088.hpp:127-136
  gx *= (0.5 * inv_sample_freq_);  // 乘以 dt/2（dt = 1/1000s = 0.001s）
  gy *= (0.5 * inv_sample_freq_);
  gz *= (0.5 * inv_sample_freq_);
  qa = q0_; qb = q1_; qc = q2_;   // 保存旧值

  q0_ += ((-qb * gx) - (qc * gy) - (q3_ * gz));
  q1_ += ((qa * gx) + (qc * gz) - (q3_ * gy));
  q2_ += ((qa * gy) - (qb * gz) + (q3_ * gx));
  q3_ += ((qa * gz) + (qb * gy) - (qc * gx));

  这是四元数微分方程的离散化：

  $$\dot{q} = \frac{1}{2} q \otimes \omega$$

  其中 ω 是角速度（三维向量看作纯四元数）。直觉上就是：每1ms，按照陀螺仪测到的角
  速度，把当前姿态旋转一点点。

  // bmi088.hpp:139-143  归一化，防止数值漂移导致四元数不再是单位四元数
  recip_norm = 1 / std::sqrt(q0_*q0_ + q1_*q1_ + q2_*q2_ + q3_*q3_);
  q0_ *= recip_norm; q1_ *= recip_norm; q2_ *= recip_norm; q3_ *= recip_norm;

  2.4 IMU 结果如何进入 TF 树（omni_infantry.cpp:237-245）

  void update_imu() {
      bmi088_.update_status();  // 运行一次Mahony，更新q0~q3

      Eigen::Quaterniond const gimbal_imu_pose{
          bmi088_.q0(), bmi088_.q1(), bmi088_.q2(), bmi088_.q3()};

      // 注意！这里取了共轭（conjugate）
      tf_->set_transform<rmcs_description::PitchLink,
  rmcs_description::OdomImu>(
          gimbal_imu_pose.conjugate());

      *gimbal_yaw_velocity_imu_   = bmi088_.gz();  // Z轴角速度 = yaw速度
      *gimbal_pitch_velocity_imu_ = bmi088_.gy();  // Y轴角速度 = pitch速度
  }

  为什么取共轭？

  IMU 测量的四元数含义是：从世界（OdomImu）坐标系旋转到机体（PitchLink）坐标系的
  变换，记作 q_world_to_body。

  但 TF 树需要的是：PitchLink 相对于 OdomImu 的变换，即
  q_body_relative_to_world。

  这两者互为逆变换，四元数的逆就是其共轭（因为单位四元数 q⁻¹ = q*）。

  第三章：TF 变换树原理

  文件：rmcs_description/include/rmcs_description/tf_description.hpp

  3.1 TF 树的结构

  BaseLink（底盘）
    └─ GimbalCenterLink（云台安装点，固定偏移）
         └─ YawLink（yaw电机转动）
              └─ PitchLink（pitch电机转动）
                   ├─ MuzzleLink（枪口）
                   ├─ CameraLink（相机）
                   └─ OdomImu（世界惯性系）← IMU四元数挂在这里

  每个节点就是一个坐标系，父子节点之间的"关节"描述了两者的相对变换。

  3.2 YawLink 关节（第81-89行）

  3.3 PitchLink 关节（第92-100行）

      using Parent = rmcs_description::YawLink;  // 父坐标系是YawLink

      }
  };

  直觉：云台上下转（pitch）就是绕 Y 轴旋转（Y轴是左右方向）。

  3.4 OdomImu 关节（第121-124行）

  template <>
  struct fast_tf::Joint<rmcs_description::OdomImu> {
      using Parent                 = rmcs_description::PitchLink;  // 
  挂在PitchLink下
      Eigen::Quaterniond transform = Eigen::Quaterniond::Identity();
      // transform 由 set_transform() 从外部直接赋值
  };

  这里最特殊：OdomImu 在物理上挂在 PitchLink 下面，但它通过 IMU
  四元数来定义自己的姿态，使得从任意坐标系 cast 到 OdomImu
  就相当于在世界绝对坐标系中表达向量。

  为什么这么设计？

  IMU 测量的是"PitchLink 相对于世界的旋转"。把这个旋转的逆存在 Joint<OdomImu>
  里，就构造出了一个"虚拟的世界坐标系节点"挂在树上。任何人只要做
  fast_tf::cast<OdomImu>(v) 就能把向量转到世界坐标系，无需了解底层细节。

  3.5 硬件如何更新 TF 树

  // omni_infantry.cpp:228-230
  tf_->set_state<GimbalCenterLink, YawLink>(gimbal_yaw_motor_.angle());
  //              ↑父             ↑子          ↑电机编码器角度

  tf_->set_state<YawLink, PitchLink>(gimbal_pitch_motor_.angle());

  每次控制周期（1ms），把电机的真实角度写进去，整棵 TF
  树就自动更新了所有坐标系之间的关系。

  3.6 fast_tf::cast 是什么

  auto v_in_odom = fast_tf::cast<OdomImu>(v_in_pitchlink, *tf_);

  这行代码的意思是：把一个在 PitchLink 坐标系里表达的向量，转换成在 OdomImu 
  坐标系里表达的向量。

  背后的数学：从 PitchLink 一路向上遍历 TF 树（PitchLink → YawLink →
  GimbalCenterLink → BaseLink），再找 OdomImu
  的路径，把所有旋转矩阵串联起来乘一遍。

  4.3 控制逻辑（第57-104行）

  第一优先级：禁用检查

  自瞄系统（视觉）输出一个在世界坐标系（OdomImu）中的目标方向向量，比如 [0.9, 
  0.1, 0.05]，表示"目标在云台正前方略偏右偏上"。这里把它包装成

  第三优先级：首次启用对齐

      return two_axis_gimbal_solver.update(TwoAxisGimbalSolver::SetToLevel());

  第一次启用控制时（云台之前是禁用的），先把云台对齐到水平前方，避免突然跳转。

  手动控制：计算增量

  // simple_gimbal_controller.cpp:76-82
  constexpr double joystick_sensitivity = 0.006;  // 手柄每个单位对应0.006 rad
  constexpr double mouse_sensitivity    = 0.5;    // 鼠标每个单位对应0.5 rad

  double yaw_shift =
      joystick_sensitivity * joystick_left_->y()   // 手柄Y轴 → yaw
      + mouse_sensitivity * mouse_velocity_->y();  // 鼠标Y轴 → yaw

  double pitch_shift =
      -joystick_sensitivity * joystick_left_->x()   // 手柄X轴（取负）→ pitch
      - mouse_sensitivity * mouse_velocity_->x();  // 鼠标X轴（取负）→ pitch

  注意负号：手柄 X 轴向右是正，但向右拨杆应该让 pitch 往下（正方向），因此取负。

  joystick_sensitivity = 0.006 的含义：手柄最大输入是 ±660（原始值归一化后约
  ±1），每 1ms 的移动量是 1 * 0.006 = 0.006 rad，相当于每秒最大旋转速度约 6 
  rad/s ≈ 344°/s。

  第五章：TwoAxisGimbalSolver 核心原理

  文件：rmcs_core/src/controller/gimbal/two_axis_gimbal_solver.hpp

  这是整个系统最复杂、最核心的部分。

  5.1 它要解决的问题

  问题：我知道目标在世界坐标系中的方向（比如 OdomImu 中的 [0.9, 0.1, 
  0.0]），我需要知道：
  - yaw 电机要转多少度误差？
  - pitch 电机要转多少度误差？

  这两个误差送给 PID，PID 输出速度指令给电机，电机转动，云台对准目标。

  5.2 update() 主函数（第103-119行）

  AngleError update(const Operation& operation) {
      update_yaw_axis();   // 步骤1：更新yaw轴方向（低通滤波）

      PitchLink::DirectionVector control_direction = operation.update(*this);
      // 步骤2：根据操作类型计算"控制方向"（在PitchLink坐标系中的目标方向向量）

      if (!control_enabled_)
          return {nan_, nan_};

      auto [control_direction_yaw_link, pitch] =
  pitch_link_to_yaw_link(control_direction);
      // 
  步骤3：把方向从PitchLink坐标系转换到YawLink坐标系，同时提取当前pitch信息

      clamp_control_direction(control_direction_yaw_link);
      // 步骤4：限制pitch角度不超出上下限

      if (!control_enabled_)
          return {nan_, nan_};

      control_direction_ =

  fast_tf::cast<OdomImu>(yaw_link_to_pitch_link(control_direction_yaw_link,
  pitch), *tf_);
      // 步骤5：把限幅后的方向存回OdomImu坐标系（供下一帧使用）

      return calculate_control_errors(control_direction_yaw_link, pitch);
      // 步骤6：计算yaw/pitch角度误差
  }

  5.3 为什么要在 PitchLink 中保存控制方向

  control_direction_ 是成员变量，类型是
  OdomImu::DirectionVector，保存的是"我希望云台指向哪里"（在世界坐标系中）。

  每帧通过 fast_tf::cast<PitchLink>(control_direction_, *tf_) 转换到当前
  PitchLink 坐标系，再叠加新的偏移量。这样做的好处是：即使底盘在运动（导致 
  PitchLink 的物理方向在变），控制方向在世界坐标系中是稳定的——人或自瞄系统只关心
  "我要打那个方向"，不关心底盘朝哪。

  5.4 SetControlShift 手动偏移（第73-97行）

  这是手动控制的核心：

  PitchLink::DirectionVector update(TwoAxisGimbalSolver& super) const override {
      PitchLink::DirectionVector dir;

      if (!super.control_enabled_) {
          // 首次启用：从正前方开始
          dir = PitchLink::DirectionVector{Eigen::Vector3d::UnitX()};
      } else {
          // 把上一帧的控制方向（OdomImu坐标系）转换到当前PitchLink坐标系
          dir = fast_tf::cast<PitchLink>(super.control_direction_, *super.tf_);
      }

      // 在PitchLink坐标系中，绕Z轴旋转yaw_shift（左右偏移）
      auto yaw_transform   = Eigen::AngleAxisd{yaw_shift_,
  Eigen::Vector3d::UnitZ()};
      // 在PitchLink坐标系中，绕Y轴旋转pitch_shift（上下偏移）
      auto pitch_transform = Eigen::AngleAxisd{pitch_shift_,
  Eigen::Vector3d::UnitY()};

      // 先yaw旋转，再pitch旋转（注意顺序：pitch * (yaw * dir)）
      return PitchLink::DirectionVector{pitch_transform * (yaw_transform *
  (*dir))};
  }

  直觉解释：
  - 把"我现在想打的方向"转到自己的坐标系里
  - 在这个坐标系里，把方向向量旋转一点点（手柄/鼠标的输入量）
  - 新的方向就是"我下一帧想打的方向"

  为什么在 PitchLink 里做旋转，而不是 OdomImu？

  在 PitchLink 里做旋转，YawLink 的 Z 轴就是旋转轴——这样 yaw
  的旋转轴永远跟随云台，符合人的操作直觉（我往左推摇杆，云台相对于自己往左转，而
  不是相对于世界往左转）。

  但等等，这里用的是 UnitZ() 和 UnitY()，是 PitchLink 的 Z/Y 轴，不是 YawLink
  的。这是个近似吗？

  不是。由于 YawLink 和 PitchLink 的唯一区别就是一个 pitch 旋转（绕 Y 轴），所以
   YawLink 的 Z 轴在 PitchLink 中会稍微偏转。代码中 update_yaw_axis()
  正是为了精确跟踪这个偏移，但在 SetControlShift 里做的是近似（直接用
  UnitZ），这在实际使用中误差很小。

  5.5 yaw 轴滤波（第124-129行）

  void update_yaw_axis() {
      // YawLink 的 Z 轴（竖直轴）在 PitchLink 坐标系中的表达
      auto yaw_axis =

  fast_tf::cast<PitchLink>(YawLink::DirectionVector{Eigen::Vector3d::UnitZ()},
  *tf_);

      // 把它转到 OdomImu（世界坐标系），做低通滤波
      *yaw_axis_filtered_ += 0.1 * (*fast_tf::cast<OdomImu>(yaw_axis, *tf_));
      yaw_axis_filtered_->normalize();
  }

  为什么要滤波？

  底盘运动时，YawLink 的 Z
  轴在世界坐标系中的方向会快速变化（底盘颠簸）。低通滤波（每帧混入 10%
  新值，保留 90% 旧值）让这个变化平滑，避免控制抖动。

  注意：*yaw_axis_filtered_ += 0.1 * new_value 这里没有乘以 0.9
  的旧值，看起来像是累积而非替换。但后面紧跟
  normalize()（归一化），所以实际效果是：方向向量每帧稍微转向新方向一点点，再归
  一化保持单位长度。这是一种简化的球面插值。

  5.6 pitch_link_to_yaw_link 坐标变换（第131-145行）

  这是最难理解的部分，需要仔细看。

  auto pitch_link_to_yaw_link(const PitchLink::DirectionVector& dir) const
      -> std::pair<YawLink::DirectionVector, Eigen::Vector2d> {

      // 获取yaw轴（竖直轴）在 PitchLink 坐标系中的方向
      auto yaw_axis = fast_tf::cast<PitchLink>(yaw_axis_filtered_, *tf_);

      // pitch 是一个二维向量：[cos(pitch_angle), sin(pitch_angle)]
      pitch = {yaw_axis->z(), yaw_axis->x()};
      pitch.normalize();

      // 坐标变换
      const auto& [x, y, z] = *dir;  // 在PitchLink中的x,y,z分量
      dir_yaw_link = {x * pitch.x() - z * pitch.y(),
                      y,
                      x * pitch.y() + z * pitch.x()};
      return result;
  }

  这里在做什么？

  YawLink 和 PitchLink 之间只差一个绕 Y 轴的 pitch 旋转。设 pitch 角为
  θ，则旋转矩阵为：

  PitchLink → YawLink 的旋转矩阵（绕Y轴旋转θ）:
  [cos θ,  0,  sin θ]   [x]   [x*cos θ + z*sin θ]
  [  0,    1,    0  ] × [y] = [       y           ]
  [-sin θ, 0,  cos θ]   [z]   [-x*sin θ + z*cos θ]

  代码里 pitch = {yaw_axis->z(), yaw_axis->x()} 正是在提取 [cos θ, sin θ]：

  - yaw_axis 是 YawLink 的 Z 轴 [0,0,1] 在 PitchLink 中的表达
  - 绕 Y 轴旋转 θ 后，[0,0,1] 变成 [sin θ, 0, cos θ]
  - 所以 yaw_axis->z() = cos θ，yaw_axis->x() = sin θ

  然后变换公式：
  dir_yaw_link.x = x * cosθ - z * sinθ   ← 对应上面矩阵第一行
  dir_yaw_link.y = y                       ← Y轴不受绕Y轴旋转影响
  dir_yaw_link.z = x * sinθ + z * cosθ   ← 对应上面矩阵第三行

  为什么不直接用 fast_tf::cast<YawLink>(dir, *tf_)？

  因为 fast_tf::cast 使用的是电机编码器的实时角度，而这里用的是经过低通滤波的
  yaw_axis_filtered_。滤波后的轴更稳定，控制更平滑。

  5.7 clamp_control_direction pitch 限幅（第154-170行）

  void clamp_control_direction(YawLink::DirectionVector& control_direction) {
      const auto& [x, y, z] = *control_direction;

      // 水平投影（在YawLink坐标系中，x-y平面是水平面）
      Eigen::Vector2d projection{x, y};
      double norm = projection.norm();
      if (norm > 0)
          projection /= norm;  // 归一化水平分量
      else {
          control_enabled_ = false;  // 方向向量指向正上/正下，无法分解yaw，禁用
          return;
      }

      // upper_limit_ = [cos(upper_angle), -sin(upper_angle)]
      // upper_limit_.y() = -sin(upper_angle) 就是"允许的最大z分量"
      if (z > upper_limit_.y())  // z太大（太仰头）
          *control_direction << upper_limit_.x() * projection, upper_limit_.y();
          // 把z限制到上限，同时保持水平方向不变
      else if (z < lower_limit_.y())  // z太小（太低头）
          *control_direction << lower_limit_.x() * projection, lower_limit_.y();
  }

  为什么 upper_limit 存的是 [cos, -sin]？

  在构造函数里：
  // two_axis_gimbal_solver.hpp:30-31
  upper_limit_(std::cos(upper_limit), -std::sin(upper_limit))
  lower_limit_(std::cos(lower_limit), -std::sin(lower_limit))

  假设 upper_limit = -0.395 rad（仰头约22°），那么：
  - cos(-0.395) ≈ 0.923 → 水平分量（x-y平面投影长度）
  - -sin(-0.395) ≈ 0.385 → z 分量（向上为正）

  这样 upper_limit_ 直接就是方向向量在 YawLink 坐标系中的最大仰角对应的
  [水平分量, z分量]，限幅时直接用就行。

  5.8 calculate_control_errors 计算角度误差（第172-183行）

  static AngleError calculate_control_errors(
      const YawLink::DirectionVector& control_direction, const Eigen::Vector2d& 
  pitch) {

      const auto& [x, y, z] = *control_direction;  // 在YawLink坐标系中的方向
      const auto& [c, s] = pitch;                   // [cos(θ), 
  sin(θ)]，θ是当前pitch角

      AngleError result;

      // yaw误差：水平面内，目标方向与X轴的夹角
      result.yaw_angle_error = std::atan2(y, x);

      // pitch误差：需要考虑当前pitch角的偏移
      double x_projected = std::sqrt(x * x + y * y);  // 水平投影长度
      result.pitch_angle_error = -std::atan2(z * c - x_projected * s,
                                             z * s + x_projected * c);
      return result;
  }

  yaw 误差很好理解：

  YawLink 坐标系中，X轴是云台正前方，Y轴是左方。
  atan2(y, x) = 目标方向在水平面的偏角
              = 目标在正前方偏左多少弧度（正值偏左，负值偏右）

  这就是 yaw 电机需要旋转的误差量。

  pitch 误差需要深入理解：

  问题：pitch 电机当前已经有一个角度 θ（电机不在零位）。我们想让 pitch
  电机额外转 Δθ，使得云台对准目标。Δθ 是多少？

  在 YawLink 坐标系中，目标方向为 [x, y, z]，水平投影为 x_proj = 
  √(x²+y²)，仰角为 α = atan2(z, x_proj)。

  但 pitch 电机的当前角度是 θ，所以需要转的误差是 Δθ = α - θ，即：

  Δθ = atan2(z, x_proj) - θ

  利用 atan2 的减法公式 atan2(a,b) - θ = atan2(a*cosθ - b*sinθ, a*sinθ + 
  b*cosθ)，展开得：

  Δθ = atan2(z*cos θ - x_proj*sin θ,
             z*sin θ + x_proj*cos θ)

  代码中的 c = cos θ，s = sin θ，前面有个负号是因为 pitch
  电机正方向的定义（仰头为负）：

  result.pitch_angle_error = -std::atan2(z*c - x_projected*s, z*s +
  x_projected*c);

  第六章：完整数据流总结

  【每1ms的控制周期】

  ① 传感器采集（CAN总线中断）
     BMI088 陀螺仪/加速度计 → int16 原始数据

  ② IMU 融合（omni_infantry.cpp:237-245）
     update_status() → Mahony算法 → 四元数 [q0,q1,q2,q3]
     → tf_->set_transform<PitchLink, OdomImu>(q.conjugate())
     → 速度输出: gz()→yaw速度, gy()→pitch速度

  ③ 电机状态更新（omni_infantry.cpp:227-230）
     yaw电机编码器 → tf_->set_state<GimbalCenterLink, YawLink>(angle)
     pitch电机编码器 → tf_->set_state<YawLink, PitchLink>(angle)

  ④ 控制器计算（simple_gimbal_controller.cpp:51-54）
     SimpleGimbalController::update()
     → calculate_angle_error()
     → TwoAxisGimbalSolver::update(SetControlShift/SetControlDirection)
        ├─ update_yaw_axis()  → 低通滤波yaw轴方向
        ├─ operation.update() → 在PitchLink中计算目标方向
        ├─ pitch_link_to_yaw_link() → 坐标变换（用当前pitch角度）
        ├─ clamp_control_direction() → pitch限幅
        └─ calculate_control_errors() → atan2计算yaw/pitch误差
     → 输出: yaw_angle_error, pitch_angle_error

  ⑤ PID控制（独立的PID节点）
     yaw_angle_error → kp*16.0 → yaw速度指令
     pitch_angle_error → kp*10.0 → pitch速度指令

  ⑥ 电机驱动（CAN发送）
     速度指令 → 电机驱动器 → 实际力矩 → 云台物理运动
