# Planner MPC 问题描述

本文描述 `tasks/auto_aim/planner/planner.cpp` 中 tinympc 求解的优化问题：目标函数、约束、输出提取，以及当前建模的局限性。

Yaw 和 pitch 通道结构完全对称，以下以单通道为例。

## 1. 问题定义

$$\min_{u_0, \dots, u_{N-1}} \sum_{k=0}^{N-1} \left[ (x_k - x_k^{\text{ref}})^\top Q\, (x_k - x_k^{\text{ref}}) + u_k^\top R\, u_k \right]$$

$$\text{s.t.} \quad x_{k+1} = A\,x_k + B\,u_k, \quad |u_k| \le u_{\max}$$

| 符号 | 含义 | 值 |
|------|------|-----|
| $x_k = [\theta_k,\; \dot\theta_k]^\top$ | 第 k 步的角度和角速度 | — |
| $x_k^{\text{ref}}$ | `get_trajectory()` 生成的参考轨迹第 k 列 | — |
| $u_k$ | 角加速度（控制输入） | — |
| $Q$ | `diag(9×10⁶, 0)` | 重惩罚角度偏差，不惩罚速度偏差 |
| $R$ | `[1]` | 轻惩罚加速度幅值 |
| $N$ | `HORIZON = 100`，覆盖 `[-0.5s, +0.5s]` | — |

**一句话概括：在满足加速度约束的前提下，让云台轨迹尽可能贴合预测瞄准轨迹，同时尽量少用加速度。**

## 2. 动力学约束

离散化的二阶积分器（`planner.cpp:110-154`）：

$$A = \begin{bmatrix} 1 & DT \\ 0 & 1 \end{bmatrix}, \quad B = \begin{bmatrix} 0 \\ DT \end{bmatrix}, \quad DT = 0.01\text{s}$$

即：

```
θ_{k+1} = θ_k + DT·ω_k
ω_{k+1} = ω_k + DT·α_k
```

## 3. 不等式约束

### 输入约束（加速度上下界）

所有时步统一：

| 通道 | 约束值（典型） |
|------|---------------|
| yaw  | $\|u_k\| \le 50$ rad/s² |
| pitch | $\|u_k\| \le 100$ rad/s² |

### 状态约束

名义上存在但实质为空：

```cpp
x_min = Eigen::MatrixXd::Constant(2, HORIZON, -1e17);
x_max = Eigen::MatrixXd::Constant(2, HORIZON,  1e17);
```

±10¹⁷ 等于无穷大，即角度和角速度不受约束。

**总结：约束结构非常简单——只有动力学 + 加速度上下界，没有角度限幅、角速度限幅、或任何其他不等式约束。**

## 4. 输出提取

MPC 求解后从第 `HALF_HORIZON`（第 50 步，对应命中时刻中点 `t=0`）处提取控制量（`planner.cpp:73-85`）：

| 输出 | 来源 | 含义 |
|------|------|------|
| `plan.yaw` | `x(0, 50) + yaw0` | 绝对 yaw 角度 |
| `plan.yaw_vel` | `x(1, 50)` | yaw 角速度 |
| `plan.yaw_acc` | `u(0, 50)` | yaw 角加速度 |
| `plan.pitch` | `x(0, 50)` | pitch 角度 |
| `plan.pitch_vel` | `x(1, 50)` | pitch 角速度 |
| `plan.pitch_acc` | `u(0, 50)` | pitch 角加速度 |

额外标志量（非 MPC 直接输出）：

| 输出 | 逻辑 | 含义 |
|------|------|------|
| `plan.control` | 目标存在且弹道可解 → `true` | 是否启用控制 |
| `plan.fire` | 第 52 步处跟踪误差 < `fire_thresh_` | 是否开火 |

### 为什么取第 50 步而不是第 0 步

100 步窗口覆盖 `[-0.5s, +0.5s]`，第 50 步对应命中时刻中点。Planner 输出的不是"立刻指向哪里"，而是"命中时刻该处于什么状态"。这组 `[角度, 角速度, 角加速度]` 作为前馈发给下位机，由下位机负责实际执行。

## 5. Q 权重的物理含义

`Q = diag(9e6, 0)` 这个设定值得注意：

- **角度权重 9×10⁶ >> 加速度权重 1**：跟踪精度压倒性优先于省力。MPC 会尽一切可能贴近参考轨迹，只在触碰加速度上限时才会妥协。
- **角速度权重 = 0**：MPC 不在乎速度本身偏离参考多少。速度是状态传播的中间量，不是优化目标。

这实际上让 MPC 退化为一个**带加速度约束的轨迹平滑器**——在约束不激活时，输出几乎等于参考轨迹本身。

## 6. 当前建模的局限性

### 6.1 MPC 没有闭环到实测状态

x₀ 取自参考轨迹首点而非 `gimbal.state()` 的实测值，因此 MPC 并不感知当前云台实际在哪里。它输出的是"目标应该怎么动"，而非"从当前状态出发该怎么追"。真正的闭环由下位机完成——下位机收到 `[yaw, yaw_vel, yaw_acc]` 后，以此作为前馈/参考去驱动电机。

### 6.2 常数加速度界是对电机的粗糙近似

对 FOC 控制的 PMSM 来说，"常数加速度界"忽略了两个物理效应：

**电机的力矩方程**：

$$\tau = K_t \cdot i_q = J \cdot \alpha + b \cdot \omega + \tau_{\text{friction}}$$

**真正的物理约束来自两处**：

| 约束来源 | 物理限制 | 对加速度的影响 |
|----------|----------|---------------|
| **电流极限** $\|i_q\| \le i_{\max}$ | 电机/驱动器的最大电流 | $\alpha_{\max} = \frac{K_t \cdot i_{\max} - b\omega}{J}$，**随速度递减** |
| **电压极限** $\|v\| \le v_{\text{bus}}$ | 母线电压有限 | 高转速时反电动势 $K_e \omega$ 吃掉电压余量，可用力矩进一步下降 |

共同构成速度-力矩包络：

```
力矩 τ
  │
  ├────────┐← 电流限制区（低速段，τ_max ≈ 常数）
  │         ╲
  │          ╲ ← 电压限制区（高速段，τ_max 随 ω 下降）
  │           ╲
  └────────────╲──── 角速度 ω
```

当前模型用 $|\alpha_k| \le \alpha_{\max}$（常数）建模，等价于：忽略阻尼项 $b \cdot \omega$、忽略电压限制、假设转动惯量 $J$ 恒定且已知。

对云台场景，如果工作转速远低于电机极限转速，这个近似可以接受。但跟踪高速旋转目标时，实际可用加速度会比模型认为的要小。

**改进方向 A：保持二阶积分器，但让约束随状态变化**

$$|\alpha_k| \le \frac{\min(K_t i_{\max},\; \tau_{\text{voltage}}(\omega_k)) - b|\omega_k|}{J}$$

需要 tinympc 支持状态相关的输入约束（或在每步线性化近似）。

**改进方向 B：改变控制输入为力矩**

$$A = \begin{bmatrix} 1 & DT \\ 0 & 1 - \frac{b}{J}DT \end{bmatrix}, \quad B = \begin{bmatrix} 0 \\ \frac{DT}{J} \end{bmatrix}$$

约束变为 $|\tau_k| \le \tau_{\max}$（低速段近似为常数就够了），阻尼效应进入动力学方程本身。改动量很小——只改 `setup_yaw_solver` / `setup_pitch_solver` 中的 A、B 矩阵和约束值。需要辨识的参数是 $J$（转动惯量）和 $b$（阻尼系数）。

## 7. MPC vs 贪心：跳变处理

当前 MPC 使用二次代价，其解不是时间最优而是在跟踪误差和加速度之间做折中。对于"追板"场景，实际需要的是"尽快到位"而非"省加速度"。

### 贪心方案

```
非跳变区间：u_k = ref_acceleration（直接微分跟踪）
跳变点附近：u_k = ±max_acc（最大制动/加速，贪心切换到下一块板）
```

对双积分器而言，从一个状态转移到另一个状态的**时间最优控制**就是 bang-bang（先全力制动，再全力加速）。在跳变点直接用最大加速度，本质上就是 bang-bang，是时间最优的。

### 关键：何时开始制动

给定当前角速度 ω 和最大减速度 a_max，制动距离是：

$$d_{\text{brake}} = \frac{\omega^2}{2 \cdot a_{\max}}$$

需要在参考轨迹上向前扫描，找到下一个跳变点，判断"当前到跳变点的角度差"是否已经小于 $d_{\text{brake}}$。如果是，就开始制动。

前提是跳变点可以可靠地从参考轨迹中检测出来——可以直接看 `get_trajectory()` 输出中相邻步的 yaw 差值是否超过阈值，或从 `armor_xyza_list()` 中判断最近装甲板编号是否发生了切换。
